Question 1: Expression System / DSL Business Problem
----------------------------------------------------
What concrete business problem did it solve, and who used it?
What was the before/after state?
What became possible once it shipped?

Your answer:
The core problem with the existing expression system implementation used in a number of our other products is that its user experience was extremely poor. Because the architecture underpinning the existing implementation was so poorly written and maintained, it was infeasible to re-use it as the expression system in our new product without compromising the user experience. Mining engineers using our software are frustrated with how hard the existing expression system implementations are to use, and we wanted to make writing expressions in our new software as painless as possible.


The existing expression system implementation systems in my company were held together with bubble-gum and sticky tape. It was a buggy mess that still has serious bugs to this day in places where it still used. It had no formal structure or testing. In most places the user was just presented with a text area where they would type in an expression and hope and pray they don't get a cryptic error message. 

The expression system that I designed and built fixes basically everything wrong with the existing one. The new expression system has:
* A robust and well-defined data-typing system
* High adaptation capabilities and is written to be product-agnostic
* Full localization and translation support for 10+ languages
* A formal programming language architecture (lexer/parser/analyzer/compiler)
* Features you would expect from an IDE or good editor (syntax highlighting, tool tips, formatting, linting, etc.)

Question 7: Strongest Performance Optimisation Example
------------------------------------------------------
What was slow?
How did you diagnose it?
What changes did you make?
What was the result in numbers?

Examples:
- runtime reduced by X%
- memory reduced by Y%
- workflow time from A to B

Your answer:
Calculations for parcel flow field calculations (these are formula calculations -- associated with the aforementioned expression system -- that apply to each parcel of material out of the pit, on each potential leg of its journey around the mine, in each potential time step). There are two key performance problems when dealing with this kind of data, fields may be dependent on one another so you need to ensure you are evaluating them in the correct order. Data can get very big (max total data points = P (number of parcels) * F (number of fields) * L (number of flow links -- transport legs) * T (number of time steps). The first of the problems is solved by arranging fields into a dependency graph, sorting the graph topologically, and evaluating fields in parallel (new field calculations are unlocked when all of their predecessors are complete). The second of the problems is a matter of analysing the expressions used for each of the fields to see how they vary in terms of flow links and time steps and minimizing the distinct number of flow-link-time-step permutations that need to be evaluated for a given field. Results are also cached in this manner. In applying the optimizations mentioned, I was able to reduce the time and space required parcel flow field calculations by 1000 times in typical client models.


Question 8: Integration Testing Framework / Synthetic Model Generator

- slow release cycles
- low confidence in scheduling workflows
- hard-to-reproduce bugs
- something else

Your answer:
* Difficulty in defining test cases. Developers were writing the same old boilerplate to write their integration tests and it was getting hard to tell which examples to follow.
* Generally poor integration coverage. Developers were scared to write integration tests because it would take too long.

The integration testing framework and synthetic model generator I crafted make it extremely easy to craft a test case that makes various configurations to a simple project, runs the solve and verifies the result. Integration tests are also a requirement when fixing bugs that are seen as a regression. The product is consequently much more stable.
